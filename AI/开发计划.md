# 从 0 开始开发计划

## 当前项目状态

- 当前仓库是一个刚初始化的 Godot 项目。
- 还没有角色、场景、脚本、资源目录和测试流程。
- 当前阶段应优先建立最小可运行战斗原型，而不是追求完整内容。

## 开发总策略

采用 `核心玩法先行 -> 快速验证 -> 再补反馈和扩展` 的方式推进。

优先级固定为：

1. 角色可控
2. 双形态成立
3. 子弹可被处理
4. 敌人可被击杀
5. 波次可运行
6. UI 和资源闭环成立
7. 反馈与手感打磨

## 阶段拆分

## Phase 0：项目骨架

目标：让项目从空工程进入“可持续开发”状态。

交付物：

- 统一目录结构
- 主场景 `Main`
- 玩家场景 `Player`
- 基础配置文档
- AI 文档目录

任务：

- 建立推荐目录：`scenes/`、`scripts/`、`resources/`、`ui/`、`audio/`、`vfx/`、`AI/`
- 确定命名规范和脚本挂载方式
- 创建主循环入口场景
- 确定输入映射

完成标准：

- Godot 能打开并运行到主场景
- 后续所有系统都有明确挂载位置

## Phase 1：玩家核心控制

目标：先把“操作是通的”做出来。

交付物：

- 角色移动
- 朝向逻辑
- 近战 / 远程状态切换
- 冲刺雏形

任务：

- 实现 WASD 八方向移动
- 鼠标位置驱动角色朝向
- 右键切换形态
- Space 触发短冲刺和无敌时间预留接口

完成标准：

- 玩家能稳定移动、切换、冲刺
- 画面中能清楚分辨当前形态

## Phase 2：战斗原型

目标：验证“近战弹反 + 远程切割”。

交付物：

- 近战普通攻击
- 普通弹斩碎 / 弹反逻辑
- 远程飞剑跟随与切割
- 剑意基础增减

任务：

- 近战扇形判定
- 子弹类型区分：普通弹、危险弹
- 普通弹被斩时的状态变化
- 飞剑最大半径、移动速度、伤害判定
- 成功处理弹幕时增加剑意

完成标准：

- 玩家能通过近战明确处理普通弹
- 玩家能通过远程明确击杀敌人
- 两种形态有显著差异

## Phase 3：敌人与波次

目标：让核心玩法进入完整战斗循环。

交付物：

- 散射兵
- 厚甲兵
- 后排施法者
- 第一版 WaveManager

任务：

- 为敌人抽象基础类：移动、攻击、受伤、死亡
- 完成三类敌人参数差异
- 按 3 分钟流程配置波次
- 实现基础刷怪点和节奏切换

完成标准：

- 单局可跑完一个 3 分钟流程
- 玩家必须切换形态才能高效通关

## Phase 4：资源与 UI

目标：建立战斗闭环和可读性。

交付物：

- 生命系统
- 剑意系统
- 绝招“剑来”雏形
- 基础 HUD

任务：

- 玩家受伤、死亡、回复基础逻辑
- 剑意获取、消耗、上限
- Q 技能清屏或大范围斩杀原型
- UI 显示生命、剑意、当前形态

完成标准：

- 玩家能理解资源来源和消耗
- HUD 足够支撑调试和试玩

## Phase 5：反馈与手感打磨

目标：把“能玩”推进到“有爽点”。

交付物：

- 命中停顿
- 屏幕震动
- 粒子与拖尾
- 基础音效

任务：

- 强化近战斩击弧光和弹反反馈
- 强化飞剑拖尾和命中特效
- 调整敌人密度、子弹速度、剑意获取速度
- 做第一轮手感自测记录

完成标准：

- 新玩家能在 1 分钟内理解规则
- 至少一个战斗片段能稳定打出爽感

## 首轮里程碑

### M1：可操作原型

- 玩家移动
- 形态切换
- 近战攻击动画占位

### M2：核心玩法成立

- 子弹可发射
- 普通弹可被近战处理
- 飞剑可切敌人

### M3：单局循环成立

- 三类敌人齐备
- WaveManager 可跑完整 3 分钟流程
- UI 与剑意闭环成立

### M4：可试玩 Demo

- 战斗反馈明显
- 参数完成第一轮调优
- 可供外部试玩与收集反馈

## 当前建议的第一周安排

### 第 1 天

- 搭目录
- 配输入映射
- 建主场景和玩家场景
- 完成移动与形态切换

### 第 2 天

- 完成近战攻击判定
- 接入基础子弹
- 完成普通弹斩碎逻辑

### 第 3 天

- 完成远程飞剑控制
- 完成飞剑切敌伤害
- 补充剑意积累原型

### 第 4 天

- 完成三类敌人基础行为
- 完成第一版刷怪和波次

### 第 5 天

- 完成 UI
- 完成绝招原型
- 进行第一轮试玩记录

## 风险控制

- 不要过早做复杂美术和演出。
- 不要在核心手感未验证前加入成长树和大量升级选项。
- 不要让近战和远程都变成“纯输出”，必须保持职责差异。
- 不要让危险弹可被随意处理，否则规则辨识会变差。

## AI 每阶段产出要求

- 每个新系统先给出节点结构建议。
- 每个脚本先定义职责，再写实现。
- 每轮迭代都补一份简短测试记录。
- 所有设计变更同步更新 `AI` 目录文档。
