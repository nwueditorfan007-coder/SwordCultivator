# 当前阶段开发计划

## 文档用途

- 本文档从 `2026-04-21` 起取代旧的“从 0 开始”启动计划，作为当前阶段的执行顺序依据。
- 目标不是重新定义项目总方向，而是把下一轮最值得做的迭代顺序、验证目标和实现边界写清楚。
- 如果其他旧文档仍保留早期启动期描述，以本文件的当前优先级为准。

## 当前项目状态

- 已有可运行的 Godot 战斗原型。
- 当前原型已经具备：
  - `WASD` 移动
  - 左键近战
  - 右键御剑点刺 / 连斩
  - 连续变形剑阵
  - 4 类常规敌人
  - 1 个 Boss 与 `Puppet / Silk` 机制
  - 基础波次与 HUD
- 当前最明显的问题已经不再是“有没有系统”，而是：
  - 剑阵按钮是否可信
  - 三形态是否一眼能懂
  - 长按维持输出时是否有操作压感
  - 敌人与 Boss 是否足够持续地产生新问题

## 当前阶段总判断

- 下一轮不要先扩更多系统，也不要先堆更多怪。
- 先修“按钮可信度”和“战场读感”，再补“持续输出的操作感”，最后再扩内容与难度。
- 如果这个顺序反过来做，试玩里很容易把“反馈问题”误判成“内容不够”。

## 当前固定优先级

1. `剑阵失败反馈 / 剑意可信度`
2. `形态切换确认 UI`
3. `持续输出的阵压手感`
4. `敌人与 Boss 的难度 / 内容扩展`

## Priority 1：剑阵失败反馈 / 剑意可信度

### 玩家当前正确感受到的问题

- 玩家会在 `剑意不足` 时继续长按剑阵。
- 玩家已经收到过失败提示，但仍然没有在当下立刻意识到“为什么没打出来”。
- 这说明问题不是“完全没有反馈”，而是反馈没有出现在玩家正在看的地方。

### 根因判断

- 当前失败反馈主要还是偏 HUD 边角语义。
- 长按持续输出时，玩家注意力在角色、准星、敌人与弹幕，不在资源数字。
- 因此“资源归零”没有被读成一次明确的动作失败，而更像“系统突然没响应”。

### 本项目标

- 让玩家在不盯数值条的前提下，也能立刻知道：
  - 是 `剑意不足`
  - 还是 `飞剑未回收`
- 让剑阵从“可能失灵的长按”变成“可信的持续输出按钮”。

### 交付物

- 发射前的低剑意预警
- `剑意不足` 与 `飞剑未回收` 的差异化失败反馈
- 更靠近角色或准星的失败提示
- 剑意归零时更明确的断流 / 收势语义

### 任务拆分

- 梳理当前所有剑阵失败原因与对应反馈通道。
- 把失败提示前移到战场核心视区，而不只停留在 HUD 染色。
- 为 `剑意不足` 与 `飞剑未回收` 补独立音色或至少独立节奏。
- 给“快空了”补前置信号，避免玩家只能在失败后才知道资源见底。
- 调整失败提示节流，避免长按时要么刷屏、要么弱到看不见。

### 验收标准

- 第一次外部试玩里，玩家在 1 到 2 次空放后就能口头说出失败原因。
- 不看数值条，也能感知“快没剑意了”和“已经没剑意了”的差别。

## Priority 2：形态切换确认 UI

### 玩家当前正确感受到的问题

- 现在的连续变形是漂亮的，但“我已经切到哪个形态”还不够稳。
- 玩家能看到图形在变，却不一定在高压战斗里立刻确认：
  - 当前已经进了 `环阵`
  - 还是还在 `环 -> 扇` 过渡
  - 或已经进入 `贯穿阵`

### 根因判断

- 当前视觉连续性已经成立，但缺少“跨过阈值时的确认语义”。
- 玩家需要的不是常驻说明书，而是短促、可信、不会盖住战场的确认提示。

### 本项目标

- 保留连续变形的签名体验。
- 补上“已经进入某个形态”的瞬时确认。
- 让三形态在高压时也能靠一眼、一耳分清。

### 交付物

- 跨阈值瞬间的轻量确认提示
- `环 / 扇 / 贯` 三形态的颜色 / 音色区分
- 仅在长按或蓄阵阶段出现的弱预览强化

### 任务拆分

- 定义三形态的确认提示语言，优先轻量而不是常驻大 UI。
- 在阈值跨越时加入一次性 pulse，而不是持续弹字说明。
- 统一形态颜色、拖尾、提示音的对应关系。
- 验证提示是否会抢走战场注意力，必要时继续减法。

### 实现边界

- 不做常驻大字模式提示。
- 不把连续变形退回成硬切模式。
- 不用额外新按键帮助玩家选型。

### 验收标准

- 外部试玩中，玩家能在战斗后复述三形态的大致近中远职责。
- 高压场景里，形态确认速度变快，但屏幕不会显得更杂。

## Priority 3：持续输出的阵压手感

### 玩家当前正确感受到的问题

- 长按维持剑阵现在有输出，但还不够“压住一股剑势”。
- 持续输出缺少逐步累积的身体负荷感，时间一长容易变平。

### 根因判断

- 当前系统有密度、有命中，但没有足够强的持续压感曲线。
- 如果直接套 FPS 式后坐力，很容易把问题从“缺操作感”做成“纯输入税”。

### 本项目标

- 不是做传统枪械后坐力，而是做一层更适合题材的 `阵压`。
- 让长按维持从“挂机输出”变成“需要压住、掌住、放掉再续”的持续控制。

### 交付物

- 重新讨论后的 `Priority 3` 原型方向
- 最小可试错参数集
- 明确哪些形态应保持稳定、哪些形态适合承担操作负荷

### 任务拆分

- 先回到当前手感基线，再重新讨论方向。
- 下一轮原型优先控制变量，不急着同时改控制层、视觉层和形态几何。
- 先确认：
  - 哪个形态应该承担持续操作负荷
  - 负荷应该来自持续顶手、单发脉冲，还是别的节奏压力
- 设计可快速开关的参数组，避免每次试错都牵连整套手感。

### 实现边界

- 不做单纯的准星乱飘。
- 不做持续累积且很难恢复的重惩罚。
- 不用更高伤害去假装“操作感提升”。

### 验收标准

- 玩家能明显感觉到“持续压阵”比现在更有身体负担和节奏管理。
- 但不会因为新机制而失去形态选择的信任感。

### 当前状态

- `2026-04-21`：已有的 `Priority 3` 原型实现已全部回退。
- 当前重新回到“先讨论、再做最小原型”的阶段，不继续沿用上一轮阵压参数。
- 后续再次进入实现前，先明确：
  - 要保护的现有手感是什么
  - 要新增的操作负荷具体来自哪里
  - 这份负荷是否真的值得玩家为它支付输入成本

### 补充专项：飞剑出入阵轨迹流动感

- 最近一轮关于飞剑轨迹的反馈已经收口为两点：
  - 当前原型方向是对的，但已经 `太花 / 太绕`
  - 单次射击时，返航阶段容易只读到黄色粒子，读不到飞剑本体
- 这项工作当前不单独抬成新优先级，而归入“手感与读感收口”范畴处理。
- 已补一份终版实现依据：
  - `docs/systems/SWORD_ARRAY_FLOW_TRAJECTORY_DESIGN.md`
- 后续实现顺序按该文档执行：
  1. 先稳住 `returning` 的本体可见性与返航可读性
  2. 再把切线起手 / 导引余韵 / 回收回卷收敛到低幅度
  3. 最后再做 `环 / 扇 / 贯` 的差异化抛光
- 当前明确禁止的方向：
  - 用更大随机扰动冒充自然感
  - 让返航粒子替代返航实体
  - 为了“灵动”牺牲发射可信度
  - 让 `贯穿阵` 也出现明显大回旋
  - 只在长按高密度场景下看轨迹，不单测单发返航
- 这一专项的最终目标不是“更炫”，而是：
  - `顺势飞出`
  - `顺势归阵`
  - `有流动感但不绕`
  - `单发返航也能看清`

## Priority 4：敌人与 Boss 的难度 / 内容扩展

### 玩家当前正确感受到的问题

- Demo 里的怪物和 Boss 套路目前偏少。
- 现有难度更多来自数值和屏幕填充，而不是来自新问题。

### 根因判断

- 常规波次目前仍主要围绕 `Shooter / Tank / Caster / Heavy` 组合。
- Boss 也仍以单个主体和有限状态循环为主。
- 如果此时只加血量、伤害或刷怪密度，玩家更容易觉得乱，而不是觉得深。

### 本项目标

- 难度提升优先来自“战斗问题种类增加”。
- 让波次和 Boss 更明确地逼玩家切换处理逻辑，而不是只比拼堆量。

### 交付物

- 1 个新的高速突脸型敌人
- 1 个新的控场 / 布场型敌人
- 现有 Boss 在关键血线的 phase package 强化
- 更明确的波次组合语法

### 任务拆分

- 新敌人先围绕“补战术问题空缺”设计，不先围绕模型数量设计。
- 常规波次从“随机刷”逐步过渡到“组合出题”。
- Boss 优先补 `70% / 30%` 血线下的问题变化，而不是先做第二只 Boss。
- 难度验证要回答的是“玩家为什么换打法”，不是“场上是不是更满了”。

### 实现边界

- 不用纯增血、纯增伤、纯增密度当作主要难度方案。
- 不急着立刻开第二只 Boss。
- 不在读感和按钮可信度还没收稳前大扩敌人规模。

### 验收标准

- 玩家能明显感到不同波次在提不同问题。
- Boss 战在血线变化后，节奏和处理优先级真的会变。

## 推荐实施顺序

### Sprint A：可信度与读感收口

- 先完成 `Priority 1`
- 再接 `Priority 2`
- 目标：先把“长按剑阵是否可信”和“我现在是什么形态”做稳

### Sprint B：持续输出手感原型

- 推进 `Priority 3`
- 目标：先明确哪个形态值得承接持续操作负荷，再做最小原型验证

### Sprint C：内容与难度扩展

- 在前两项验证通过后推进 `Priority 4`
- 目标：确认内容扩展是在放大已有优点，而不是掩盖基础读感问题

## 当前明确不优先做的事

- 不先做大体量常驻 UI 改版
- 不先做第二只 Boss
- 不先做更多数值层成长系统
- 不先靠堆怪密度解决“内容不够”
- 不先把持续输出手感做成传统 FPS 后坐力

## 下一轮试玩必须回答的问题

1. 玩家是否能在空放后的第一时间说出失败原因？
2. 玩家是否能在不暂停思考的情况下确认当前形态？
3. `环 / 扇 / 贯` 的职责读感是否比之前更稳定？
4. 当前基线下，持续输出真正缺的是哪一种操作负荷，它应该先落在哪个形态上？
5. 波次和 Boss 的新内容，是在提新问题，还是只是更乱？

## AI 协作要求

- 后续如果再调整优先级，先更新本文件，再扩实现。
- 讨论手感问题时，先写玩家体感，再写根因，再写最小原型。
- 任何新内容扩展都要说明：它是在补“可信度 / 读感 / 操作压感 / 问题种类”中的哪一项。
