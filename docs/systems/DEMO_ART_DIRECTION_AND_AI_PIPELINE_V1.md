# 剑修 Demo 美术方向与 AI 产出框架 V1

- 日期：`2026-04-22`
- 性质：实现导向设计文档
- 目标：为当前 demo 建立统一的美术方向、视觉分层和 AI 协作生产框架，使后续的概念图、HUD、特效、背景、图形资产和 Godot 落地都围绕同一套标准推进。

## 1. 这份文档解决什么问题

当前项目的核心战斗识别已经存在：

- 连续变形剑阵
- 可见飞剑库存
- 御剑 / 近战 / Boss 机制切换
- 偏“单屏仪式战场”的战斗结构

但当前画面仍处于“高可读原型”阶段，距离参考图里的那种完成度，缺的不是单个特效，而是整套视觉总控：

- 战场缺少仪式化舞台感
- UI 仍像功能面板，而不是阵图的一部分
- 背景、战斗、HUD、后处理还不是同一门视觉语言
- 当前 `draw_*` 几何已经能说明机制，但还没形成最终审美

因此这份文档不回答“某个 shader 怎么写”，而是先回答：

- 这个 demo 最终要长成什么样
- 哪些东西适合交给 AI 先大量产出
- 哪些东西必须人工或技术美术强把关
- Godot 当前工程最合理的落地路径是什么


## 2. 核心判断

### 2.1 目标不是“更炫”，而是“更像一张活着的阵图”

参考图最打人的地方，不是 bloom 更大、粒子更多，而是以下五件事同时成立：

- 战场像法阵舞台，而不是普通关卡地面
- 飞剑是唯一真正高亮、真正快速、真正锐利的主体
- 敌人与机制体被压缩成少量高辨识符号
- UI 与战场不是分离的两层，而像同一件法器的不同部分
- 背景很静，爆发很短，留白很多

### 2.2 当前项目已经有可保留的底子

当前项目最不该丢的东西：

- 连续变形的视觉手感
- 飞剑作为可见实体的库存读感
- 以战斗几何驱动反馈而不是纯装饰
- 以读感优先而不是盲目堆料

这些都是正确方向，不需要为了追参考图而推翻。

### 2.3 真正的缺口是视觉总控

当前主要缺口不是某一条拖尾，而是以下层级尚未建立：

- `Scene Base`：背景舞台
- `Tactical Glyph`：阵环、符标、机制图形
- `Combat VFX`：战斗反馈
- `UI Ornament`：HUD 装饰框架
- `Composite`：整体合成和亮度控制


## 3. 建议的目标风格

## 3.1 结论

建议采用：

- `图4 / 图5` 的骨架
- `图1 / 图6` 的气质

也就是：

- 结构上更克制、更可做
- 气质上仍然保留玄夜星图、道法仪式感、飞剑神性

### 3.2 为什么不是直接追最华丽版本

如果直接追最华丽的 painterly 版本，当前 demo 会立刻遇到几个问题：

- 资产量暴增
- 合成难度暴增
- Godot 原型的迭代速度明显下降
- 很容易把战场可读性吃掉

而 `图4 / 图5` 这类更克制的版本，有更高的成本收益比：

- 更容易与当前几何原型结合
- 更适合先做一张高完成度英雄镜头
- 更容易让 AI 批量产出基础资产
- 更容易在后续逐步向 `图1 / 图6` 的华丽度推进


## 4. 视觉圣经

## 4.1 一句话美术命题

`玄夜天幕中的剑修法阵战斗。`

不是：

- 写实场景战斗
- 高饱和仙侠页游
- 科幻霓虹 UI

而是：

- 深夜
- 星图
- 水墨余痕
- 金线秩序
- 冷白飞剑
- 血红异术

## 4.2 构图原则

- 中心必须有“阵心”或“剑势源点”
- 玩家系主体亮度最高
- 主要攻击矢量尽量清晰、单向、利落
- 战场外围保持空，让中心更显神性
- 大书法字、云纹、水纹、圆环只能做氛围，不抢战术主体

## 4.3 颜色规则

### 基底色

- 深蓝黑：背景主色
- 暗青灰：次级环境层

### 秩序色

- 古金：阵图、边框、标题、HUD 装饰

### 玩家色

- 冷白：核心亮点
- 青蓝：飞剑边缘、拖尾、能量走向

### 敌对色

- 绯红：敌弹、警告、机制威胁
- 暗红：次级危险体

### 禁止项

- 大面积高饱和蓝紫
- 大面积彩虹渐变
- 让金色和飞剑同时抢主亮度

## 4.4 亮度预算

画面里只允许三类高亮：

- 飞剑本体
- 关键命中 / 宣告瞬间
- 极少量 HUD 核心强调

其余元素都必须让位：

- 背景纹样偏暗
- 书法字偏灰
- 阵图环线偏细
- UI 边框偏淡金
- 敌方常驻符标亮度低于飞剑

## 4.5 纹样与材质

建议重复使用的视觉语素：

- 同心圆
- 轨道弧线
- 星点
- 金线边框
- 水墨云纹
- 淡书法残字
- 几何符标

不要大量引入：

- 写实石砖地面
- 高频贴图噪声
- 大块实体金属 UI 面板
- 过多卡牌式弹窗

## 4.6 字体与文案风格

建议采用双字体体系：

- 标题 / 功法名 / 模式名：书法或碑刻感中文字体
- 数值 / 操作提示 / 小字：清晰稳定的现代中文 UI 字体

原则：

- 题字负责气质
- 小字负责可读
- 不要让所有字都走书法体


## 5. 场景分层框架

## 5.1 Scene Base

职责：

- 建立“玄夜法阵舞台”
- 提供整体氛围
- 负责静态美术完成度

包含：

- 深色底幕
- 水墨烟云
- 淡书法大字
- 轻星点
- 微弱纹样

要求：

- 静
- 暗
- 不能抢战斗读感

## 5.2 Tactical Glyph

职责：

- 提供战斗语义的图形骨架
- 让阵型、机制、敌方威胁更像术式而不是 debug 图

包含：

- 阵心圆环
- 星轨 / 弧轨
- Boss 机制符标
- 丝线 / 锚点 / 锁定圈
- 目标红色符印

要求：

- 读得懂
- 轻量
- 不替代真正的战斗本体

## 5.3 Combat VFX

职责：

- 负责宣告、运动、接触、收束
- 延续现有 `State / Declaration / Motion / Contact / Resolve` 体系

要求：

- 锚定真实几何
- 快
- 脆
- 不糊

## 5.4 UI Ornament

职责：

- 把 HUD 从功能条升级成法器构件

包含：

- 左上资源区重做为金线 + 图腾 + 细发光
- 上中标题与边饰
- 左右竖排箴言 / 道法短句
- 底部技能盘和按键图标
- 模式图标与资源点亮状态

要求：

- 仍然服务战斗
- 不做厚重面板
- 不和中心战场争焦点

## 5.5 Composite

职责：

- 统一最终画面质感

包含：

- 轻 bloom
- 弱 vignette
- 局部 blur
- 微颗粒
- 受控的 chroma / distortion

要求：

- 最终调味，不是主菜
- 一定受亮度预算约束


## 6. 对 Godot 当前工程的落地判断

## 6.1 当前路线可以继续用

当前项目基于 `Node2D + draw_*` 的几何绘制路线没有问题。

原因：

- 剑阵连续变形本来就适合几何驱动
- 战斗判定与视觉锚点已经比较接近
- 原型迭代速度快
- 可读性强

不建议为了追参考图，立刻把核心战斗全改成重材质或复杂节点树。

## 6.2 需要升级的不是“求解”，而是“合成”

更合理的升级方向是：

1. 保留战斗几何求解
2. 将背景、HUD、敌方符标逐步资产化
3. 增加独立的屏幕级合成层
4. 逐步拆出视觉总控脚本

## 6.3 推荐的渲染拆分

建议后续收敛为四段：

1. `background pass`
2. `combat pass`
3. `ui pass`
4. `composite pass`

短期不要求一次性完成，但这是目标结构。


## 7. AI 产出边界判断

## 7.1 结论

不是“全部都能无脑交给 AI”，而是：

- 大部分前期和中期产出，AI 都可以高效参与
- 很多基础资产，AI 可以直接出第一版
- 最终统一、筛选、取舍、合成和引擎适配，仍然必须有人强把关

更准确地说：

- `AI 很适合做第一版和变体量产`
- `技术美术 / 美术负责人必须负责统一审美和最后 20%`

## 7.2 AI 强项区

以下内容很适合让 AI 直接产出：

- 风格探索图
- Moodboard
- 颜色方案变体
- 背景概念图
- 水墨云纹 / 星图纹样 / 书法残字贴图
- HUD 概念稿
- 按键图标、资源图标、阵图装饰件
- Boss 符标、丝线术式、敌方警告符印
- 飞剑拖尾参考帧
- 技能宣告图形参考
- 过场海报感单帧
- 用于贴图的素材切片
- shader 思路草案
- 规范文档初稿
- 分镜和任务拆解

## 7.3 AI 可做，但不能直接信最终结果

以下内容 AI 可以做很多，但必须强筛选：

- 飞剑本体设计
- UI 定稿
- 敌人符号系统
- 背景主图
- 字体搭配
- 光效图形语言
- 局部动画关键帧参考
- Godot shader 代码草案
- 后处理参数建议

风险是：

- 好看但不统一
- 有气质但不可读
- 单图成立，实战一糊就全废
- 代码能跑，但不适合项目当前结构

## 7.4 当前不建议完全交给 AI 的部分

以下部分必须由人主导最终决策：

- 最终视觉方向裁剪
- 战斗读感与信息量控制
- 亮度预算
- UI 层级优先级
- Godot 内实际合成参数
- 性能与实现约束
- 玩法信息和美术表现之间的取舍

本质原因不是 AI 不会生成，而是这些决定属于：

- 审美总控
- 战斗可读性总控
- 工程落地总控


## 8. 最推荐的 AI 协作方式

## 8.1 不要一句话让 AI“做整套美术”

这是最低效的方式。

更有效的方式是先把任务拆到明确产物：

- 一页 HUD 方案
- 一页背景方案
- 一组敌方红色符标
- 一组飞剑拖尾帧
- 一组阵图金纹边饰

## 8.2 每次只固定一个变量

例如：

- 同一套构图，只换配色
- 同一套配色，只换 HUD 语言
- 同一套 HUD，只换装饰密度

这样 AI 才能真正成为可控生产力，而不是随机抽卡。

## 8.3 先做母版，再做批量

推荐顺序：

1. 先定一张英雄镜头母版
2. 用母版反推视觉规则
3. 再批量生成边框、图标、纹样、变体

不要先量产一堆零件，再试图拼成同一套风格。

## 8.4 AI 产出必须走“选、修、压”三步

### 选

- 先筛掉不符合项目命题的图

### 修

- 人工或 AI 二次迭代，修统一性和可读性

### 压

- 压亮度
- 压细节
- 压噪声

当前这类风格最忌讳“每层都很努力”。

## 10. 推荐的首轮制作顺序

## 10.1 第一阶段：定母版

目标：

- 先做出一张“这就是它”的英雄镜头

产物：

- 单张主视觉
- 色板
- HUD 基础样式
- 背景基调

验收问题：

- 飞剑是不是全场唯一主角
- UI 有没有法器感
- 背景是不是静
- 红色威胁是不是够清楚

## 10.2 第二阶段：拆零件

目标：

- 把母版拆成可复用资产

产物：

- 边框件
- 角饰
- 阵图件
- 资源图标
- 技能图标
- 敌方符标
- 背景贴图层

## 10.3 第三阶段：进 Godot

目标：

- 把“好看单图”变成“能打的战斗画面”

产物：

- 背景底层
- HUD 初版
- 战场符标层
- 初版 composite

## 10.4 第四阶段：战斗特效统一化

目标：

- 让飞剑、命中、时停、强化全部说同一种语言

依赖：

- 延续现有 `COMBAT_VFX_ART_FRAMEWORK_V1`


## 11. 当前阶段最该避免的反模式

- 一开始就要求 AI 直接生成完整可用游戏界面并马上实装
- 还没定母版，就先做大量散件
- 让背景、HUD、特效各走一套风格
- 让所有元素都发光
- 让书法字、阵图、飞剑、敌弹同时抢高亮
- 让 AI 直接决定战斗信息层级
- 在没有亮度预算的前提下先上 bloom


## 12. 现阶段的执行建议

当前最合理的推进方式是：

1. 先由 AI 产出主视觉母版和 HUD 方向稿
2. 再由人裁剪出风格规则
3. 再批量让 AI 生产素材零件
4. 再在 Godot 中进行合成、取舍和战斗读感校准

换句话说：

- AI 可以成为主产能
- 但不能成为最终审美裁判


## 13. 结论

如果问题是：

`这些内容，全部都能让 AI 进行工作产出吗？`

更准确的答案是：

`绝大部分都可以让 AI 产出第一版，其中相当一部分可以直接进入制作；但决定项目最终是否高级、是否统一、是否能打，仍然取决于人对风格、可读性和落地的强控制。`

对当前项目，最优策略不是“AI 替代人”，而是：

- 用 AI 负责高频探索和批量产出
- 用技术美术负责方向统一和引擎落地
- 用战斗设计标准约束最终信息量

这套组合，才最有机会把当前原型推进到参考图那个层级。
