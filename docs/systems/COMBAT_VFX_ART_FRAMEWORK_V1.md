# 战斗特效美术框架 V1

- 日期：`2026-04-21`
- 性质：实现导向设计文档
- 目标：为当前 Godot 原型建立一套可持续扩展的战斗特效框架，使“御剑时停、出鞘、剑阵、命中、资源反馈、Boss 机制”都能在同一语言下实现，而不是每次单点堆特效。

## 1. 这份框架解决什么问题

当前项目已经具备明确的战斗机制，但视觉反馈仍处于“事件存在，声明不够强；有粒子和震屏，但语言未统一”的阶段。

最近围绕“御剑出鞘一闪”的迭代表明，问题不只是某一条 flare 画得好不好，而是：

- 一个特效同时承担了太多职责
- 触发时机和动作结构没有被拆开
- 缺少统一的特效分层和锚点规则

结果就是：

- 有些效果热闹，但不好读
- 有些效果看起来会“晚半拍”
- 新需求一来，只能继续在 `main.gd` / `game_renderer.gd` 里临时补判断

这份框架的目标不是先做更复杂的美术，而是先建立：

- 哪类效果负责声明动作
- 哪类效果负责表达运动
- 哪类效果负责说明状态
- 哪类效果负责说明命中和资源变化

只有分工清楚，效果才能越做越稳。

## 2. 当前工程约束

当前项目的视觉实现有明确边界：

- 主战场是 `Node2D + draw_*` 手绘渲染，不是材质驱动的场景树特效
- 当前已经有轻量粒子、拖尾、震屏和状态文案
- Boss 走独立绘制逻辑，但仍然属于同一套战场画面
- 当前没有真正的后处理链，也没有现成的统一 VFX 控制器

因此 V1 框架应遵循：

- 优先使用当前 `draw_*` 管线和轻量运行时数据
- 不依赖复杂 shader 才能成立
- 不要求把所有效果拆成节点
- 优先建立“事件驱动 + 分层绘制 + 统一语义”

如果后续效果量明显增大，再考虑升级到更重的 `SubViewport + shader` 路线。

## 3. 总原则

### 3.1 声明和运动分离

一个动作开始时，需要先让玩家知道“事情发生了”；之后再让玩家读到“它往哪去了”。

因此：

- `声明型特效` 负责瞬时爆发和状态切换确认
- `运动型特效` 负责拖影、残影、方向、流向

不要让一条 flare 同时承担“我出鞘了”和“我朝这里飞过去了”。

### 3.2 状态和动作分离

“时停”是战场状态，不是某个单点动作的附属火花。

因此：

- 时停的主表达应是全场状态变化
- 出鞘、点刺、连斩的主表达应是动作声明

不要把“时停感”全部寄希望于出鞘那一下。

### 3.3 玩家可读性优先于装饰性

这是动作弹幕项目，不是观赏型演出项目。

所有效果先问三个问题：

- 它是否强化了输入反馈
- 它是否强化了战场读感
- 它是否会遮挡敌弹、敌人或飞剑位置

如果第三项风险成立，效果强度必须让步。

### 3.4 同类职责共用一套语言

不要为每个系统发明一套全新的视觉语法。

项目里应该重复出现同一批基础视觉语素：

- 线
- 弧
- 带
- 点状高亮
- 冷暖对比
- 短促爆发

不同技能只是在这些语素上变体，而不是完全换一门语言。

### 3.5 特效必须锚定真实几何

剑阵、御剑、Boss 丝线、飞剑发射都已经有明确几何结果。

后续所有轨迹类特效，应尽量消费已有几何和方向结果，而不是自己再猜。

否则视觉会和判定分离。

## 4. 五层特效框架

### 4.1 状态层 State Layer

职责：

- 声明战场进入某个状态
- 在较长时间内维持统一氛围

典型对象：

- 御剑时停
- 强化剑技
- Boss 脆弱期

典型手段：

- 全场去饱和 / 恢复
- 冷白 wash
- 环境亮度变化
- UI 条高亮

规则：

- 只负责“状态已变”
- 不负责方向、命中、节奏点
- 通常影响敌方、背景、Boss、敌弹
- 默认不压暗玩家阵营

### 4.2 声明层 Declaration Layer

职责：

- 在动作提交瞬间给出明确确认

典型对象：

- 御剑出鞘
- 近战斩出
- 剑阵发射起手
- 技能释放失败

典型手段：

- 短促 flare
- 一帧亮点
- 局部高亮弧线
- 极短白热闪

规则：

- 生命周期极短
- 位置基本稳定，不做明显位移
- 强调“发生了”，不强调“飞过去了”

### 4.3 运动层 Motion Layer

职责：

- 告诉玩家攻击或武器怎么运动

典型对象：

- 御剑点刺残影
- 连斩拖尾
- 飞剑出击拖尾
- 返航飞剑尾迹

典型手段：

- 残影
- 拖尾
- 条带
- 分帧 ghost

规则：

- 几何必须跟真实运动方向一致
- 透明衰减必须快
- 不要比武器本体更抢眼

### 4.4 接触层 Contact Layer

职责：

- 表达“打中了 / 弹开了 / 切断了 / 清掉了”

典型对象：

- 近战命中敌人
- 弹反敌弹
- 飞剑穿身
- 切断 Boss 丝线
- 环阵清弹

典型手段：

- 命中火花
- 切断碎光
- 小幅震屏
- 局部 hit flash

规则：

- 锚定接触点
- 持续时间短
- 可以强，但不能糊

### 4.5 收束层 Resolve Layer

职责：

- 表达资源兑现、系统结果、战斗阶段变化

典型对象：

- 吸收成功
- 飞剑补阵
- 强化开启
- 资源不足失败反馈

典型手段：

- 向心收束粒子
- 槽位点亮
- UI 条脉冲
- 回收补阵闪

规则：

- 让玩家知道“这个动作产生了什么战果”
- 这层比接触层更偏结果，不偏打击

## 5. 建议的运行时结构

### 5.1 不要继续把所有效果硬编码进单个逻辑分支

建议在当前项目内建立统一的效果运行时数据层。

V1 先不做节点化，直接用轻量数组即可：

- `screen_effects`
- `declaration_effects`
- `motion_effects`
- `contact_effects`
- `resolve_effects`

每个 effect 至少包含：

- `type`
- `time`
- `duration`
- `position`
- `direction`
- `color`
- `intensity`
- `space`
- `owner`

### 5.2 `owner` 语义必须进入框架

所有效果都要有归属语义：

- `player`
- `player_weapon`
- `enemy`
- `boss`
- `world`
- `ui`

这会直接决定：

- 是否受时停去色影响
- 是否叠加 wash
- 是否允许压过 HUD

### 5.3 `space` 语义必须进入框架

所有效果要声明坐标空间：

- `world`
- `screen`
- `ui`

例如：

- 时停 wash 是 `screen`
- 出鞘 flare 是 `world`
- 失败提示文案是 `ui`

### 5.4 渲染顺序必须固定

建议的绘制顺序：

1. 背景 / 网格
2. 状态层环境影响
3. 敌方单位 / Boss / 敌弹
4. 玩家系运动层之前的世界粒子
5. 玩家 / 御剑 / 飞剑
6. 声明层
7. 运动层
8. 接触层
9. 收束层
10. HUD / UI

不要每次需要某个效果时临时决定它画在什么前后。

## 6. 颜色语义框架

项目当前已经有良好的基础色：

- 玩家：绿
- 御剑 / 远程：青蓝
- 近战：金黄
- 敌人：红系
- Boss：紫
- 丝线：白 / 绯红

建议在此基础上补三套“效果语义色”：

### 6.1 冷态控制色

用于：

- 时停
- 读招
- 空间冻结

特征：

- 低饱和冷白
- 微蓝，不要纯蓝滤镜

### 6.2 热态爆发色

用于：

- 出鞘 flare
- 近战重击
- 强化起爆

特征：

- 白热核心
- 暖金边缘

### 6.3 切断 / 裂解色

用于：

- 丝线切断
- 飞剑穿身
- 清弹

特征：

- 明亮白芯
- 青白或红白碎裂边

原则：

- 状态多用冷态
- 声明多用热态
- 接触多用裂解态

## 7. 动画时长与曲线原则

### 7.1 瞬时声明

适用于：

- flare
- 微闪
- hit spark

建议时长：

- `0.03s ~ 0.08s`

原则：

- 起得快
- 消得更快
- 不做平移

### 7.2 短运动残影

适用于：

- 御剑出鞘残影
- 连斩尾迹

建议时长：

- `0.05s ~ 0.12s`

原则：

- 跟真实方向
- 衰减快
- 不抢主体

### 7.3 状态恢复

适用于：

- 时停颜色恢复
- 强化态收束

原则：

- 完全跟随系统状态时间轴
- 不另起自己的表演节奏

## 8. 御剑出鞘的正确表达

这部分是当前最直接相关的落地建议。

### 8.1 问题判断

当前“会位移的 flare”之所以不好，是因为它把职责混在了一起：

- 它既想声明“出鞘发生了”
- 又想表达“往哪里出去了”

结果看起来就会：

- 有延时感
- 像是特效晚到半拍
- 像一条自己飞出去的光，不像刀出鞘

### 8.2 正确拆法

御剑出鞘建议拆成三件事：

1. `输入即时反馈`
2. `动作提交声明`
3. `武器运动残影`

### 8.3 推荐结构

#### A. 输入即时反馈

触发时机：

- `右键按下`

表现：

- 一个极小的局部热点击
- 强度很弱，只做“我按到了”

作用：

- 解决右键按下到点刺/连斩真正进入状态之间的空窗

#### B. 动作提交声明

触发时机：

- 真正进入 `POINT_STRIKE` 或 `SLICING`

表现：

- 原地瞬发 flare
- 不位移
- 轴向为“出鞘方向法线主导 + 少量水平偏置”
- 时长极短

作用：

- 告诉玩家“状态切换已提交”

#### C. 武器运动残影

触发时机：

- 御剑开始运动后的前几帧

表现：

- 2 到 3 帧残影
- 跟随真实出鞘方向
- 比 flare 弱

作用：

- 告诉玩家“是往这边切出去的”

### 8.4 关键原则

- flare 不移动
- 方向感交给残影
- 时停感交给状态层

这三者不要互相抢活。

## 9. 项目内其他系统的效果模板

### 9.1 近战挥剑

- 声明层：斩击弧光
- 运动层：短弧尾
- 接触层：命中火花 / 弹反爆点
- 收束层：剑意获取脉冲

### 9.2 飞剑点刺 / 连斩

- 声明层：出鞘 flare
- 运动层：刀身残影、线性拖尾
- 接触层：穿身切裂
- 状态层：时停去色

### 9.3 剑阵发射

- 声明层：起阵亮点
- 运动层：阵型拖尾 / 批次出锋
- 接触层：命中 burst
- 收束层：回阵补位反馈

### 9.4 Boss 丝线

- 状态层：主丝线威胁高亮
- 声明层：切断瞬时亮点
- 接触层：断裂碎闪
- 收束层：Boss 脆弱暴露

### 9.5 资源不足

- 声明层：失败提示轻闪
- 收束层：对应资源条脉冲

不要为“失败”做大粒子，避免和成功释放混淆。

## 10. 实现建议

### 10.1 V1 不建议立即上复杂后处理

当前阶段更推荐：

- 继续使用 `draw_*`
- 把效果事件结构化
- 把颜色语义统一
- 把绘制顺序固定

### 10.2 当效果种类继续增长时，再做 `game_vfx_controller.gd`

建议的升级路径：

1. 当前 `main.gd` 先持有 effect arrays
2. 当 effect 种类明显增多时，再抽出 `game_vfx_controller.gd`
3. 由 `GameRenderer` 专注绘制，`GameVfxController` 专注更新和发射

### 10.3 什么时候再考虑 shader

只有在以下需求同时成立时，再考虑重后处理路线：

- 需要更强的屏幕级时停风格
- 需要 bloom / chromatic aberration / distortion
- 当前 `draw_*` 路线已经无法满足表现

在此之前，不建议让项目提前背上复杂管线。

## 11. 下一步的最优先落地顺序

### 第一阶段

- 把“御剑出鞘 flare”改回原地瞬发声明
- 增加很轻的按下微反馈
- 为点刺 / 连斩补一层短残影

### 第二阶段

- 统一近战 / 御剑 / 剑阵的声明层语法
- 统一接触层命中 burst 语法

### 第三阶段

- 将强化态、Boss 脆弱期、时停全部接到状态层语义
- 统一颜色恢复和 wash 规则

### 第四阶段

- 抽离 `game_vfx_controller.gd`
- 建立通用 effect emit API

## 12. 反模式

以下做法应尽量避免：

- 一条 flare 同时承担声明、方向、位移、命中
- 用长时间移动的光效表达瞬时出鞘
- 让时停把玩家阵营一起压灰
- 每个系统单独发明一套颜色和运动语言
- 特效层级顺序依赖临时 if 判断

## 13. 结论

这个项目最适合的，不是一套“越做越复杂的堆料式特效系统”，而是一套：

- 分工明确
- 语义统一
- 几何锚定清晰
- 可持续扩展

的战斗特效框架。

对于当前最关键的御剑出鞘问题，结论已经很明确：

- 回到“原地瞬发声明”
- 用残影表达方向
- 用状态层表达时停

这条路比继续打磨“会移动的 flare”更稳，也更符合当前项目的底层结构。
